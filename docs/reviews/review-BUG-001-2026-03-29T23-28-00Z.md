## ⚠️ Issues

- [ ] Shared cached fixture projects are now backed by one mutable file path and one cached ESLint instance per declaration shape, so helper calls are no longer isolated. If any of these suites start using concurrent tests or if a future refactor causes overlapping helper calls, writes to the shared `example.spec.ts` file can race and produce cross-test contamination or nondeterministic diagnostics. Affected references: `fixtureProjectsByDeclaration` and cached project reuse in [src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts](../../src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts#L36), [src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts](../../src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts#L181), [src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts](../../src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts#L229), [src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts](../../src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts#L231), [tests/support/prefer-vitest-incremental-casts-rule-test-helpers.ts](../../tests/support/prefer-vitest-incremental-casts-rule-test-helpers.ts#L34), [tests/support/prefer-vitest-incremental-casts-rule-test-helpers.ts](../../tests/support/prefer-vitest-incremental-casts-rule-test-helpers.ts#L150), [tests/support/prefer-vitest-incremental-casts-rule-test-helpers.ts](../../tests/support/prefer-vitest-incremental-casts-rule-test-helpers.ts#L189), and the `afterAll` sharing in [src/testing/prefer-vitest-incremental-casts/rule.spec.ts](../../src/testing/prefer-vitest-incremental-casts/rule.spec.ts#L6), [src/testing/prefer-vitest-incremental-casts/rule.guardrails.spec.ts](../../src/testing/prefer-vitest-incremental-casts/rule.guardrails.spec.ts#L11), [src/testing/prefer-vitest-incremental-casts/rule.input-shapes.spec.ts](../../src/testing/prefer-vitest-incremental-casts/rule.input-shapes.spec.ts#L9), and [src/testing/prefer-vitest-incremental-casts/rule.signature-fallback.spec.ts](../../src/testing/prefer-vitest-incremental-casts/rule.signature-fallback.spec.ts#L9). Severity: Medium.

- [ ] The rejection-path helper spec resets modules and dynamically re-imports a fresh helper module, but suite cleanup still calls `cleanupTemporaryDirectories` from the original import. Any temporary directories or cached ESLint instances created by the re-imported module are outside that original module state, so this test can leak temp projects and cached state until process exit. Affected references: dynamic import in [src/testing/prefer-vitest-incremental-casts/rule-test-helpers.spec.ts](../../src/testing/prefer-vitest-incremental-casts/rule-test-helpers.spec.ts#L36), suite cleanup in [src/testing/prefer-vitest-incremental-casts/rule-test-helpers.spec.ts](../../src/testing/prefer-vitest-incremental-casts/rule-test-helpers.spec.ts#L50), and module-local cache/cleanup state in [src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts](../../src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts#L36), [src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts](../../src/testing/prefer-vitest-incremental-casts/rule-test-helpers.ts#L202). Severity: Medium.

---

## 💡 Suggestions

- Keep the performance win, but restore per-call file isolation. Reusing the ESLint project and tsconfig is enough; the source-under-test should use unique filenames or a per-run scratch file to avoid shared mutable state.
- In the rejection-path spec, either clean up through the re-imported helper instance or avoid module re-instantiation for this case so the cleanup function always targets the same module-local caches it created.
