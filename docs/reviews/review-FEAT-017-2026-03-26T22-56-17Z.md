## ⚠️ Issues

- [ ] `consistent-barrel-files` now keeps module-level cache and de-duplication state across rule invocations, which can return stale filesystem results in long-lived lint processes or suppress re-reporting after files change. Severity: Medium
- [ ] `barrel-files-exports-only` is hard-wired to the default `index` stem, so repositories using custom barrel names through `consistent-barrel-files.allowedNames` or `no-reexports-outside-barrels.allowedBarrelNames` will not have their actual barrel files checked for purity. Severity: Medium
- [ ] The new e2e coverage depends on import-only `.spec.ts` wrappers because Vitest only includes `tests/**/*.spec.ts`, while the real test content remains in `.test.ts` files. This is easy to forget and can silently skip future e2e additions. Severity: Low

---

## 💡 Suggestions

- Reset or localize the `consistent-barrel-files` cache per lint run instead of keeping it at module scope.
- Decide whether `barrel-files-exports-only` should share the same barrel-name model as the other barrel rules; if yes, introduce a consistent option or shared configuration strategy.
- Normalize the e2e naming convention so the files Vitest executes are the files developers edit.
